# JWT 鉴权机制

## 一、为什么需要鉴权？

HTTP 是**无状态**的——服务器不知道两次请求是不是同一个人。所以需要鉴权机制来识别用户身份。

### 常见鉴权方案对比

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| **Cookie + Session** | 服务端存 Session，客户端存 Cookie | 有状态，服务端有压力，不好跨域 |
| **JWT** | 服务端签发 Token，客户端保存并携带 | 无状态，易扩展，适合分布式 |
| **OAuth 2.0** | 第三方授权（微信/GitHub 登录） | 适合第三方登录场景 |

## 二、JWT 是什么？

JWT（JSON Web Token）= 一个加密签名的 **JSON 字符串**，用来在客户端和服务端之间安全地传递身份信息。

> 一句话：服务器给你一张"身份证"（Token），你每次请求都带着，服务器验证签名就知道你是谁。

### JWT 的结构

一个 JWT 长这样：

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsImV4cCI6MTcwMH0.abc123signature
```

由三部分组成，用 `.` 分隔：

```
Header.Payload.Signature
 头部  . 载荷  .  签名
```

| 部分 | 内容 | 编码方式 |
|------|------|---------|
| **Header** | 算法类型 + Token 类型 | Base64 编码 |
| **Payload** | 用户数据（userId、过期时间等） | Base64 编码 |
| **Signature** | 防篡改签名 | HMAC / RSA 加密 |

```js
// Header
{ "alg": "HS256", "typ": "JWT" }

// Payload
{ "userId": 1, "username": "张三", "exp": 1700000000 }

// Signature
HMACSHA256(base64(Header) + "." + base64(Payload), 密钥)
```

> **注意**：Payload 只是 Base64 编码，**不是加密**！任何人都能解码看到内容。JWT 靠的是 Signature 防篡改，不是防偷看。所以不要放密码等敏感信息。

## 三、JWT 认证流程

```
① 用户登录：POST /login { username, password }
        ↓
② 服务端验证账号密码 → 正确 → 生成 JWT 返回给客户端
        ↓
③ 客户端保存 JWT（localStorage / Cookie）
        ↓
④ 之后每次请求都在 Header 中携带：
   Authorization: Bearer eyJhbGci...
        ↓
⑤ 服务端收到请求 → 验证 JWT 签名 + 是否过期
   ├─ 有效 → 放行，处理请求
   └─ 无效 → 返回 401 未授权
```

## 四、代码实现

### 安装依赖

```bash
npm install jsonwebtoken
```

### 生成和验证 Token

```js
const jwt = require('jsonwebtoken');

const SECRET = 'my-secret-key';  // 密钥，要保密

// ✅ 生成 Token
function generateToken(user) {
  return jwt.sign(
    { userId: user.id, username: user.username },  // payload
    SECRET,
    { expiresIn: '7d' }  // 7天过期
  );
}

// ✅ 验证 Token
function verifyToken(token) {
  try {
    const decoded = jwt.verify(token, SECRET);
    return decoded;  // { userId: 1, username: '张三', iat: ..., exp: ... }
  } catch (err) {
    return null;     // 无效或过期
  }
}
```

### 登录接口

```js
app.post('/login', async (req, res) => {
  const { username, password } = req.body;

  // 1. 查数据库验证用户
  const user = await User.findOne({ username });
  if (!user || !comparePassword(password, user.password)) {
    return res.status(401).json({ message: '账号或密码错误' });
  }

  // 2. 生成 Token 返回
  const token = generateToken(user);
  res.json({ token });
});
```

### 鉴权中间件

```js
function authMiddleware(req, res, next) {
  // 1. 从 Header 取 Token
  const authHeader = req.headers.authorization;
  if (!authHeader || !authHeader.startsWith('Bearer ')) {
    return res.status(401).json({ message: '未提供 Token' });
  }

  const token = authHeader.split(' ')[1];

  // 2. 验证 Token
  const decoded = verifyToken(token);
  if (!decoded) {
    return res.status(401).json({ message: 'Token 无效或已过期' });
  }

  // 3. 把用户信息挂到 req 上，后续路由可以用
  req.user = decoded;
  next();
}

// 使用：需要登录才能访问的接口
app.get('/api/profile', authMiddleware, (req, res) => {
  res.json({ userId: req.user.userId, username: req.user.username });
});
```

## 五、Token 刷新机制

Token 总会过期，常见的续期方案：

### 双 Token 方案（推荐）

```
accessToken  —— 短期（如 15分钟），用于请求接口
refreshToken —— 长期（如 7天），用于刷新 accessToken
```

```js
// 登录时返回两个 Token
app.post('/login', (req, res) => {
  const accessToken = jwt.sign(payload, SECRET, { expiresIn: '15m' });
  const refreshToken = jwt.sign(payload, REFRESH_SECRET, { expiresIn: '7d' });
  res.json({ accessToken, refreshToken });
});

// 刷新接口
app.post('/refresh', (req, res) => {
  const { refreshToken } = req.body;
  try {
    const decoded = jwt.verify(refreshToken, REFRESH_SECRET);
    const newAccessToken = jwt.sign(
      { userId: decoded.userId },
      SECRET,
      { expiresIn: '15m' }
    );
    res.json({ accessToken: newAccessToken });
  } catch {
    res.status(401).json({ message: 'refreshToken 过期，请重新登录' });
  }
});
```

## 六、高频面试题

### Q1：JWT 和 Session 的区别？

| 对比 | JWT | Session |
|------|-----|---------|
| 存储位置 | 客户端（localStorage/Cookie） | 服务端（内存/Redis） |
| 状态 | 无状态 | 有状态 |
| 跨域 | 天然支持（放 Header 里） | 依赖 Cookie，跨域麻烦 |
| 扩展性 | 好（分布式不需要共享状态） | 差（多台服务器要共享 Session） |
| 安全性 | Token 泄露难以主动失效 | 可以服务端主动销毁 |

### Q2：JWT 的缺点？

- **无法主动失效**：签发后只能等过期（解决：维护黑名单 / 用 Redis 存有效 Token）
- **Payload 不加密**：敏感信息不要放里面
- **Token 较大**：每次请求都要携带，比 Session ID 大

### Q3：Token 存在哪里更安全？

| 位置 | 优缺点 |
|------|--------|
| `localStorage` | 方便，但有 XSS 风险 |
| `Cookie`（HttpOnly） | 防 XSS，但有 CSRF 风险 |
| 内存（变量） | 最安全，但刷新页面丢失 |

> 推荐：accessToken 存内存，refreshToken 存 HttpOnly Cookie。
