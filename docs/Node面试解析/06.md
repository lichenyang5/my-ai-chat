# Node.js 性能与性能优化

## 一、如何衡量 Node.js 性能？

### 关键指标

| 指标 | 含义 | 怎么看 |
|------|------|--------|
| **QPS/TPS** | 每秒处理请求数 | 压测工具（ab、wrk、autocannon） |
| **响应时间** | 从请求到响应的耗时 | P50、P95、P99 |
| **内存使用** | 堆内存占用 | `process.memoryUsage()` |
| **CPU 使用率** | CPU 占用百分比 | `os.cpus()`、监控工具 |
| **事件循环延迟** | 事件循环被阻塞的时间 | `monitorEventLoopDelay` |

### 查看内存使用

```js
const mem = process.memoryUsage();
console.log({
  rss: `${(mem.rss / 1024 / 1024).toFixed(2)} MB`,          // 总内存
  heapTotal: `${(mem.heapTotal / 1024 / 1024).toFixed(2)} MB`, // 堆总量
  heapUsed: `${(mem.heapUsed / 1024 / 1024).toFixed(2)} MB`,  // 堆已用
  external: `${(mem.external / 1024 / 1024).toFixed(2)} MB`   // C++ 对象
});
```

## 二、常见性能问题

| 问题 | 表现 | 原因 |
|------|------|------|
| **内存泄漏** | 内存持续增长不释放 | 全局变量、未清除的定时器、闭包引用 |
| **CPU 阻塞** | 响应变慢、事件循环卡住 | 同步操作、大循环、复杂计算 |
| **连接数爆满** | 新请求被拒绝 | 连接未释放、数据库连接池太小 |
| **GC 频繁** | 响应时间不稳定（抖动） | 大量临时对象、堆内存接近上限 |

## 三、性能优化策略

### 1. 代码层面

#### 避免同步操作

```js
// ❌ 同步读文件（阻塞事件循环）
const data = fs.readFileSync('/big-file.txt');

// ✅ 异步读文件
const data = await fs.promises.readFile('/big-file.txt');

// ✅ 大文件用 Stream
fs.createReadStream('/big-file.txt').pipe(res);
```

#### 避免内存泄漏

```js
// ❌ 全局数组不断增长
const cache = [];
app.get('/api', (req, res) => {
  cache.push(req.body);  // 永远不清理 → 内存泄漏
});

// ✅ 使用 LRU 缓存，自动淘汰旧数据
const LRU = require('lru-cache');
const cache = new LRU({ max: 500 });
```

#### 善用缓存

```js
// 内存缓存
const cache = new Map();

async function getUser(id) {
  if (cache.has(id)) return cache.get(id);

  const user = await db.query('SELECT * FROM users WHERE id = ?', [id]);
  cache.set(id, user);
  return user;
}

// Redis 缓存（更适合生产环境）
const redis = require('redis');
const client = redis.createClient();

async function getUserCached(id) {
  const cached = await client.get(`user:${id}`);
  if (cached) return JSON.parse(cached);

  const user = await db.query('SELECT * FROM users WHERE id = ?', [id]);
  await client.set(`user:${id}`, JSON.stringify(user), 'EX', 3600);  // 1小时过期
  return user;
}
```

### 2. 架构层面

#### 使用 Cluster 多进程

```js
const cluster = require('cluster');
const os = require('os');

if (cluster.isMaster) {
  // 主进程：按 CPU 核数 fork 子进程
  const cpuCount = os.cpus().length;
  for (let i = 0; i < cpuCount; i++) {
    cluster.fork();
  }

  cluster.on('exit', (worker) => {
    console.log(`Worker ${worker.process.pid} 挂了，重启一个`);
    cluster.fork();
  });
} else {
  // 子进程：跑业务代码
  const app = require('./app');
  app.listen(3000);
}
```

> 或者直接用 **PM2**：`pm2 start app.js -i max`

#### Worker Threads 处理 CPU 密集

```js
const { Worker, isMainThread, parentPort } = require('worker_threads');

if (isMainThread) {
  // 主线程：创建 Worker
  const worker = new Worker(__filename);
  worker.postMessage({ number: 40 });
  worker.on('message', (result) => {
    console.log('计算结果:', result);
  });
} else {
  // 工作线程：执行耗时计算
  parentPort.on('message', ({ number }) => {
    const result = fibonacci(number);
    parentPort.postMessage(result);
  });
}

function fibonacci(n) {
  if (n <= 1) return n;
  return fibonacci(n - 1) + fibonacci(n - 2);
}
```

### 3. 数据库层面

| 优化方向 | 做法 |
|---------|------|
| **连接池** | 使用连接池，避免频繁创建/销毁连接 |
| **索引** | 给查询频繁的字段加索引 |
| **分页** | 大量数据必须分页，不要一次全查 |
| **N+1 问题** | 用 JOIN 或批量查询代替循环查询 |
| **读写分离** | 主库写、从库读 |

```js
// ❌ N+1 查询（查10个用户，循环10次查订单）
for (const user of users) {
  user.orders = await db.query('SELECT * FROM orders WHERE user_id = ?', [user.id]);
}

// ✅ 批量查询（1次搞定）
const userIds = users.map(u => u.id);
const orders = await db.query('SELECT * FROM orders WHERE user_id IN (?)', [userIds]);
```

### 4. 网络层面

| 优化方向 | 做法 |
|---------|------|
| **Gzip 压缩** | `compression` 中间件，减小传输体积 |
| **HTTP/2** | 多路复用，减少连接数 |
| **CDN 静态资源** | 图片/CSS/JS 走 CDN |
| **Keep-Alive** | 复用 TCP 连接 |
| **负载均衡** | Nginx 反向代理 + 多实例 |

```js

```

## 五、高频面试题

### Q1：Node.js 如何处理高并发？

1. **事件驱动 + 非阻塞 I/O**：单线程处理请求，I/O 操作交给底层线程池
2. **Cluster 多进程**：利用多核 CPU
3. **反向代理 + 负载均衡**：Nginx 分发请求到多个 Node 实例
4. **缓存**：Redis 缓存热点数据，减少数据库压力

### Q2：如何排查内存泄漏？

1. `process.memoryUsage()` 定时打印堆内存变化
2. `node --inspect` → Chrome DevTools → Memory 面板
3. 对比两次堆快照，找出增长的对象
4. 常见原因：全局变量、未清理的定时器/事件监听、闭包引用
