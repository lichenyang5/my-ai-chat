# HTTP 与 HTTPS

## 一、HTTP 是什么？

HTTP（HyperText Transfer Protocol）= **超文本传输协议**

> 一句话：浏览器和服务器之间"说话"的规则。你在浏览器输入网址，浏览器用 HTTP 向服务器要数据。

**特点**：
- **无状态**：服务器不记得上次请求是谁（所以才需要 Cookie/Session）
- **无连接**（HTTP/1.0）：请求完就断开；HTTP/1.1 起默认长连接（keep-alive）
- **基于请求-响应模型**：客户端发请求，服务器回响应
- **明文传输**：数据不加密（不安全）

## 二、HTTP 版本演进

| 版本 | 关键改进 |
|------|---------|
| **HTTP/1.0** | 每次请求建一个 TCP 连接，用完关闭 |
| **HTTP/1.1** | 长连接（keep-alive）、管道化、缓存策略 |
| **HTTP/2** | 多路复用、头部压缩、服务器推送、二进制帧 |
| **HTTP/3** | 基于 **QUIC**（UDP），解决队头阻塞 |

### HTTP/1.1 vs HTTP/2 核心区别

```
HTTP/1.1（串行排队）          HTTP/2（多路复用）
┌──────────────┐            ┌──────────────┐
│ 请求1 → 响应1 │            │ 请求1 ─┐     │
│ 请求2 → 响应2 │            │ 请求2 ─┼─ 共用│
│ 请求3 → 响应3 │            │ 请求3 ─┘ 一个 │
│  一个一个排队等 │            │   TCP 连接   │
└──────────────┘            └──────────────┘
```

## 三、HTTP 请求与响应

### 请求报文结构

```
GET /index.html HTTP/1.1          ← 请求行（方法 路径 版本）
Host: www.example.com             ← 请求头
Content-Type: application/json
                                  ← 空行
{"name": "test"}                  ← 请求体（GET 一般没有）
```

### 常见请求方法

| 方法 | 作用 | 幂等 | 有请求体 |
|------|------|------|---------|
| GET | 获取资源 | ✅ | ❌ |
| POST | 提交数据/创建资源 | ❌ | ✅ |
| PUT | 替换整个资源 | ✅ | ✅ |
| PATCH | 部分更新资源 | ❌ | ✅ |
| DELETE | 删除资源 | ✅ | ❌ |

### 常见状态码

| 分类 | 含义 | 常见 |
|------|------|------|
| **1xx** | 信息提示 | 101 切换协议 |
| **2xx** | 成功 | 200 OK、201 Created、204 No Content |
| **3xx** | 重定向 | 301 永久重定向、302 临时重定向、304 未修改（用缓存） |
| **4xx** | 客户端错误 | 400 Bad Request、401 未授权、403 禁止、404 Not Found |
| **5xx** | 服务器错误 | 500 Internal Error、502 Bad Gateway、503 Service Unavailable |

## 四、HTTPS 是什么？

HTTPS = HTTP + **SSL/TLS 加密层**

> 一句话：给 HTTP 穿上"防弹衣"，数据加密传输，防窃听、防篡改、防伪装。

```
HTTP:   应用层（HTTP）→ 传输层（TCP）
HTTPS:  应用层（HTTP）→ 安全层（SSL/TLS）→ 传输层（TCP）
```

**端口区别**：HTTP 默认 80，HTTPS 默认 443

## 五、加密方式

### 1. 对称加密

> 一把钥匙，加密解密用同一个密钥

```
明文 ─→ [密钥A加密] ─→ 密文 ─→ [密钥A解密] ─→ 明文
```

- **优点**：速度快
- **缺点**：密钥怎么安全传给对方？（密钥分发问题）
- **常见算法**：AES、DES

### 2. 非对称加密

> 两把钥匙——公钥加密，私钥解密（或反过来用于签名）

```
明文 ─→ [公钥加密] ─→ 密文 ─→ [私钥解密] ─→ 明文
```

- **优点**：不需要传私钥，安全
- **缺点**：速度慢（比对称加密慢 100+ 倍）
- **常见算法**：RSA、ECC

### 3. HTTPS 的做法：混合加密

> 用**非对称加密**安全地传递一个**对称密钥**，之后就用这个对称密钥加密通信。兼顾安全和速度。

## 六、HTTPS 握手过程（TLS 1.2）

```
客户端                                  服务端
  |                                       |
  | ① Client Hello                        |
  | （支持的TLS版本、加密套件、随机数A）──────→|
  |                                       |
  | ② Server Hello                        |
  |←──（选择的加密套件、随机数B、数字证书）    |
  |                                       |
  | ③ 客户端验证证书                        |
  |   - 证书是否可信（CA链验证）             |
  |   - 证书是否过期                        |
  |   - 域名是否匹配                        |
  |                                       |
  | ④ 客户端生成预主密钥(Pre-Master Secret)  |
  |   用服务端公钥加密后发送           ──────→|
  |                                       |
  | ⑤ 双方用 随机数A + 随机数B + 预主密钥    |
  |   各自计算出相同的对称密钥               |
  |                                       |
  |      ✅ 之后用对称密钥加密通信            |
```

## 七、HTTP vs HTTPS 对比

| 对比项 | HTTP | HTTPS |
|--------|------|-------|
| 安全性 | 明文，不安全 | 加密，安全 |
| 端口 | 80 | 443 |
| 证书 | 不需要 | 需要 CA 证书 |
| 速度 | 较快 | 稍慢（多了握手） |
| SEO | 无加分 | 搜索引擎优先 |

## 八、高频面试题

### Q1：GET 和 POST 的区别？

| 区别 | GET | POST |
|------|-----|------|
| 参数位置 | URL 上（?key=value） | 请求体里 |
| 安全性 | 参数暴露在 URL，不安全 | 相对安全 |
| 长度限制 | URL 有长度限制（浏览器限制） | 一般无限制 |
| 缓存 | 会被浏览器缓存 | 不会缓存 |
| 幂等 | 幂等（多次请求结果相同） | 非幂等 |
| 后退/刷新 | 无害 | 会重新提交 |

### Q2：HTTPS 一定安全吗？

不是 100% 安全：
- 证书可能被伪造（需要验证 CA）
- 中间人攻击（MITM）可以在用户信任了伪造证书的情况下窃听
- 服务端本身被入侵，加密也没用

### Q3：为什么要用混合加密？

- 纯对称加密：密钥传输不安全
- 纯非对称加密：太慢
- 混合使用：非对称加密只用来传密钥（一次），之后全程用对称加密（快）
