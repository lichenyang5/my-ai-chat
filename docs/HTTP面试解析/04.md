# DNS 与 CDN

## 一、DNS 是什么？

DNS（Domain Name System）= **域名系统**

> 一句话：互联网的"电话簿"——把你输入的域名（www.baidu.com）翻译成 IP 地址（39.156.66.18）。

因为人记域名方便，机器认 IP 地址方便，DNS 就是中间的翻译官。

## 二、DNS 解析过程

当你在浏览器输入 `www.example.com`：

```
① 浏览器缓存 → 有就直接用
      ↓ 没有
② 操作系统缓存（hosts 文件） → 有就直接用
      ↓ 没有
③ 本地 DNS 服务器（运营商提供）→ 有缓存就返回
      ↓ 没有
④ 根域名服务器 → "去问 .com 的服务器"
      ↓
⑤ 顶级域名服务器（.com）→ "去问 example.com 的服务器"
      ↓
⑥ 权威域名服务器（example.com）→ 返回 IP：93.184.216.34
      ↓
⑦ 本地 DNS 缓存结果，返回给浏览器
```

### DNS 查询方式

| 方式 | 说明 |
|------|------|
| **递归查询** | 客户端问本地 DNS，本地 DNS 帮你一路问到底，最终返回结果 |
| **迭代查询** | 本地 DNS 问根服务器，根说"你去问那个"，本地 DNS 再去问，一步步自己跑 |

> 实际中：客户端 → 本地DNS 是**递归**，本地DNS → 各级服务器是**迭代**。

## 三、DNS 记录类型

| 类型 | 作用 | 示例 |
|------|------|------|
| **A** | 域名 → IPv4 | example.com → 93.184.216.34 |
| **AAAA** | 域名 → IPv6 | example.com → 2606:2800:... |
| **CNAME** | 别名，指向另一个域名 | www.example.com → example.com |
| **MX** | 邮件服务器 | example.com → mail.example.com |
| **TXT** | 文本记录（验证用） | 域名所有权验证 |
| **NS** | 指定域名服务器 | example.com → ns1.example.com |

## 四、CDN 是什么？

CDN（Content Delivery Network）= **内容分发网络**

> 一句话：在全国/全球部署很多"分身"服务器，用户访问时自动从最近的节点拿数据，**更快**。

```
没有 CDN：
用户（广州）───────── 长距离 ──────────→ 源站（北京）

有 CDN：
用户（广州）──→ CDN节点（广州）  ← 缓存了源站的内容
                   快！
```

## 五、CDN 工作原理

```
用户输入 www.example.com
        |
        ↓
① DNS 解析 → CNAME 指向 CDN 的域名
        ↓
② CDN 的 DNS 智能解析 → 根据用户 IP、网络状况选最优节点
        ↓
③ 返回最近 CDN 节点的 IP
        ↓
④ 用户访问该 CDN 节点
        ↓
⑤ CDN 节点有缓存？
   ├─ 有 → 直接返回（快）
   └─ 没有 → 回源站拿数据 → 缓存后返回给用户
```

## 六、CDN 的负载均衡

CDN 如何决定把用户请求分到哪个节点？用的就是**负载均衡**。

### 负载均衡的层级

| 层级 | 方式 | 说明 |
|------|------|------|
| **DNS 级别** | 智能 DNS 解析 | 根据用户地理位置/运营商，返回不同 CDN 节点 IP |
| **HTTP 级别** | 重定向 | 服务器返回 302，让浏览器重新请求另一个地址 |
| **IP 级别** | 负载均衡器（LVS/Nginx） | 在同一入口后面分发到多个服务器 |

### 常见负载均衡策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **轮询** | 挨个分配，1→2→3→1→2→3 | 服务器性能相同 |
| **加权轮询** | 性能好的多分一点 | 服务器性能不同 |
| **IP 哈希** | 同一 IP 总是分到同一服务器 | 需要会话保持 |
| **最少连接** | 谁当前连接少就分给谁 | 请求处理时间差异大 |
| **地理位置** | 就近分配 | CDN 场景 |

## 七、高频面试题

### Q1：浏览器输入 URL 到页面展示，经历了什么？

```
① DNS 解析：域名 → IP
② TCP 连接：三次握手
③ 发送 HTTP 请求
④ 服务器处理请求，返回响应
⑤ 浏览器解析 HTML → 构建 DOM 树
⑥ 解析 CSS → 构建 CSSOM 树
⑦ DOM + CSSOM → 渲染树（Render Tree）
⑧ 布局（Layout）→ 计算位置大小
⑨ 绘制（Paint）→ 渲染到屏幕
⑩ TCP 四次挥手断开连接
```

### Q2：DNS 用的是 TCP 还是 UDP？

**大多数情况用 UDP**（快、DNS 查询数据小，一个包就够了）。

但当响应数据超过 512 字节（如区域传输），会切换到 **TCP**。

### Q3：CDN 缓存过期了怎么办？

- CDN 节点的缓存有 TTL（生存时间）
- 过期后下一次请求会**回源**（向源站重新获取最新数据）
- 也可以主动**刷新 CDN 缓存**（通过 CDN 控制台/API 清除）

### Q4：CDN 能加速动态内容吗？

- CDN 主要加速**静态资源**（图片、CSS、JS、视频）
- 动态内容可以通过**边缘计算**或**动态加速（DSA）**优化路由链路来加速
- 动态内容不做缓存，但可以优化传输路径

### Q5：为什么有时候清了浏览器缓存还是访问到旧页面？

很可能是 **CDN 缓存**还没过期。解决办法：
- 等 CDN 缓存 TTL 到期
- 在 CDN 控制台手动刷新
- 给资源加版本号（如 `app.js?v=2`）
